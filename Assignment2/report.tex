\section{Task 1}
The data structures I used for this part were numpy array for the weights and the x input features. This was used to make doing dot products and any other linear algebra operations faster because numpy is highly optimized. I also used a list to keep track of the losses of the losses during each of the epochs. The algorithms that were used were Stochastic Gradient Descent (SGD), Hinge loss, and L2 Regularization. The margin between the sample and its possible label is calculated. It does this for every sample by multiplying the true label and the dot product of the label and the weight plus the bias. This value given is used for the hinge loss function. the hinge loss function checks if the points is correctly classified and outside of the margin. If it is outside of the margin then it updates the weights by shrinking. If it is inside or the weight is wrong then it updates weights to both shrink the vector and move the decision boundary to better classify the current point. We use L2 Regularization to prevent overfitting of the data by penalizing the squared magnitude of the weights. This makes it so we try and maximize the margin to the hyperplane without overfitting the data.

The functions that were implemented in the class were init which will initialize the basic values of the LinearSVC. It can be given the learning rate, number of iterations, a random state, and a regularization parameter for use in L2 regularization. We have have a fit function that will be given the set of features and the true labels. This function will train the model based on the provided set of data. The net input function computes the preactivation value for a given input and the last is the predict function that takes a given input and makes a prediction for the given sample.

There is a lot of interesting things that can be seen from these loss convergence graphs. The way I am going to compare them is first by the number of samples and then see how the changing dimensions of the data effects it. In the first column we have 100 samples and we can see how the dimension of the data effects how fast the convergence happens. When $D=10$ we see that the loss convergence is relatively slow compared to when $D=50\text{ or } 100$. It takes more than 1000 iterations to reach a plateau, and when $D=50$ it takes less than 200 iterations, and the fastest is when D=100 it happens almost instantly. When we increase the number of samples to 500 we see this trend continue. We see that when $D=10$ we do not reach a plateau by the time we have 1000 iterations and we are at a higher loss than the other graphs. When $D=50$ we are still decreasing the slope so we would need more iterations to see the whole value of the loss but we can see that it is better than 10 and worse than 100. When we have $D=100$ we see that again the drop is almost instant and we reach a plateau at a zero loss. When we increase the number of samples to 5000 we see that that both $D=10 \text{ and } 50$ never get close to zero as they plateau at a loss of $0.06$ but with $D=100$ we are still decreasing at 1000 iterations but starting to plateau. \\

The observation that can be seen with is that when we increase the dimensions (meaning increase the features) that the model is able to find the decision boundary much more easily but each of the individual operations is more expense because of the increased vector size. The reason that increasing the number of dimensions works in this case is because the data is perfectly linearly separable but in other cases where it is not this would be a bad thing and make it slower. We also see as we increase N the learning rate the the curve becomes noisier. When testing the code the I needed to decrease the learning rate by a lot in order for the curve to become smoother so it was less effected by the changes. We see a the opposite thing when we increase the size of number of samples. When we have a small value of n (like 100) the model can be over fitted quickly leading to the flat line we see in some graphs. When n is large there are more constraints and edge cases. Since the learning rate is constant the small changes needed correctly satisfy the 5000 points of data causing the noise to increase. Another thing we see is that when the number of samples is very low and the number of dimensions is high we see that the model drops to 0 very fast and then is flat for the rest of the time. This means that with a low number of samples the number of iterations can be lowered by a lot in order to save time.

The reason why the graphs for each of these look so different is because of the definition of the primal and dual problem and how they are different. With the primal problem we are focusing on the weights and the bias directly. We know that the graphs in figure \ref{fig:loss_prime} are very smooth. This is because the primal problem is very good at handling convex optimization problems. When we change the weights slightly the loss changes predictably. Primal depends on how many dimensions we have for how complicated the task is going to be. If we only have a few features then it is able to find the optimal line very quickly and not change from that place. \\
The dual problem is different from primal. It assigns a variable to every data point and tries to find which of the points are the support vectors. We know that in figure \ref{fig:loss_dual} that for almost all of the data sets we have a lot of visible noise in the graph. This happens because the algorithm for dual will update the variable for each data point one at a time causing there to be some noise in the loss. As the algorithm goes through each of the different points, changing one point might slight throw off the balance of the other points. This makes the graph have the up and down lines in the convergence line before we each a minimum value. The complexity depends on the number of samples. That is why in table \ref{tab:time-cost_dual} dual starts to struggle as the number of samples increases.
